## Iteration 8 — US-008: Build game logic hook (useCarLoanGame)
- Created `useCarLoanGame.ts` with: handleChoice (process scenario option), score computation, resetGame
- Each round: monthly interest = (baseRate/12) × effectMultiplier, car depreciates 1.5%, loan reduced by (payment - interest)
- Interest effect multipliers: increase=1.5×, decrease=0.5×, neutral=1.0×
- Speed = 100 - (interestBurden × 200), clamped at 0. interestBurden = totalInterestPaid / originalLoan
- Repossession check: remainingLoan > carCurrentValue × 1.5 → game over early with F grade
- Grade by interestRatio (totalInterest / loanAmount): S (≤5%), A (≤10%), B (≤18%), C (≤28%), F (>28% or repossessed)
- lastFeedback state exposes the Hebrew feedback text from the chosen option for the screen to display
- Returns: state, currentScenario, lastFeedback, handleChoice, score, resetGame
- Files affected: `src/features/chapter-1-content/simulations/useCarLoanGame.ts` (new)
- Learnings for next cycles:
  - Scenario-based hook pattern (like useBudgetGame): round index in state, scenario via useMemo, advance on choice
  - Interest effect modifies the monthly interest rate multiplier, not the base rate itself
  - lastFeedback as separate useState (not in CarLoanState) keeps state type clean per PRD types
  - Speed computation inversely proportional to interest burden — 200 factor ensures visible slowdown
  - resetGame must reset both state and lastFeedback
  - All money values rounded to integers with Math.round for display consistency

## Iteration 18 — US-018: Build game logic hook (useShoppingCart)

- Items presented one at a time via currentIndex in state, currentItem derived via useMemo
- addToCart: adds item to cart, tracks totalSpent, increments essentialsCollected (if essential) or trapsFallen (if trap)

- Over-budget penalty: budgetScore linearly decreases from 20 to 0 based on overspend ratio
- moneyWasted calculated from trap items in cart for score screen display
- Returns: state, currentItem, addToCart, skipItem, score, resetGame
- Files affected: `src/features/chapter-1-content/simulations/useShoppingCart.ts` (new)
- Learnings for next cycles:
  - Hook pattern: two actions (add/skip) instead of single handler — screen needs two buttons
  - advance extracted as plain function (not useCallback) since it's only called within setState updaters
  - Budget alternatives treated neutrally — not essential and not trap, so adding them costs money but no penalty/bonus
  - No lastFeedback state needed — trap explanation popup will be handled by screen component checking currentItem.trapExplanation after addToCart

## Iteration 23 — US-023: Build game logic hook (useEmergencyFund)
- Created `useEmergencyFund.ts` with: handleSavingsChoice (process month), score computation, resetGame
- Uses InternalState extending EmergencyFundState with outstandingLoan for synchronized state updates (avoids dual-setState sync issues)
- Each month flow: (1) apply savings or handle layoff expenses, (2) handle emergency event, (3) apply loan interest, (4) advance month
- Layoff handling: months 8-9 have zero income — expenses come from fund, forced loan if insufficient
- Emergency absorption: if fund ≥ event cost → absorb (deduct from fund). Otherwise → forced loan for shortfall
- Loan interest: 12% annual = 1% monthly, compounding on outstanding loan balance each month
- MonthResult feedback type exported for screen use: savingsAdded, event, absorbed, loanTaken, expenseShortfall
- Grade: S (no loans, no interest), A (≤1 missed + <₪500 interest), B (≤2 missed), C (≤3 missed), F (>3 missed)
- Returns: state, currentEvent, isLayoffMonth, lastMonthResult, handleSavingsChoice, score, resetGame
- Files affected: `src/features/chapter-1-content/simulations/useEmergencyFund.ts` (new)
- Learnings for next cycles:
  - Combined InternalState pattern avoids dual-useState sync issues — cleaner than nested setOutstandingLoan/setState
  - Public state derived via useMemo destructuring (omits outstandingLoan)
  - setLastMonthResult called inside setInternal updater is safe (React batches these)
  - Screen (US-024) should use: savingsOptions for choice buttons, currentEvent for emergency display, lastMonthResult for feedback phase, isLayoffMonth for special UI
  - Layoff event (cost=0) is counted as "handled" since it has no direct cost — the financial impact comes from zero income in months 8-9
