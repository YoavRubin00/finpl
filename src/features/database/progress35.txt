## Iteration 4 — US-004: Sync user_profiles with useAuthStore
- Created `src/db/sync/syncUserProfile.ts` with two exports:
  - `upsertUserProfile(authId, data)` — insert or update via `onConflictDoUpdate` on `authId` unique constraint
  - `fetchUserProfile(authId)` — select by authId, returns row or null
- Hooked `upsertUserProfile` into `useAuthStore.signIn` as fire-and-forget (`.catch(() => {})`)
- Uses email as authId (the unique identifier from the auth flow)
- No new TS errors introduced
- Files affected: `src/db/sync/syncUserProfile.ts` (new), `src/features/auth/useAuthStore.ts` (modified)
- Learnings for next cycles:
  - Patterns: `onConflictDoUpdate` with `target` pointing to a unique column is the Drizzle upsert pattern
  - Patterns: Fire-and-forget with `.catch(() => {})` keeps UI non-blocking per PRD technical notes
  - Patterns: `eq()` from `drizzle-orm` for WHERE clauses
  - Traps: `authId` is the user identifier (email-based), not a UUID — userProfiles.id is the internal UUID PK
  - Helpful context: UpsertUserProfileData interface covers all mutable columns; timestamps are auto-managed

## Iteration 5 — US-005: Sync module_progress with useChapterStore
- Created `src/db/sync/syncModuleProgress.ts` with two exports:
  - `upsertModuleProgress(authId, data)` — resolves authId → UUID via userProfiles, then upserts into module_progress using `(userId, moduleId)` composite unique constraint
  - `fetchModuleProgress(authId)` — resolves authId → UUID, then selects all module_progress rows for that user
- Helper `resolveUserId(authId)` looks up userProfiles.id by authId (email) — needed because module_progress.userId is a UUID FK, not the email
- Hooked `upsertModuleProgress` into `useChapterStore.completeModule` as fire-and-forget (`.catch(() => {})`)
- Passes quiz results (correct count, total attempts) and xpEarned from the store
- No new TS errors introduced
- Files affected: `src/db/sync/syncModuleProgress.ts` (new), `src/features/chapter-1-content/useChapterStore.ts` (modified), `src/features/database/PRD35_NeonDBIntegration.md` (updated)
- Learnings for next cycles:
  - Patterns: module_progress uses composite unique on (userId, moduleId) — pass both to `onConflictDoUpdate.target` as array
  - Patterns: Need a `resolveUserId(authId)` helper since child tables FK to userProfiles.id (UUID), not authId (email)
  - Patterns: Same fire-and-forget pattern as US-004 — `.catch(() => {})` keeps UI non-blocking
  - Traps: Must resolve email → UUID before inserting into child tables; direct email won't work as userId
  - Helpful context: `useAuthStore.getState().email` is the authId used across sync functions

## Iteration 6 — US-006: Sync inventory with useEconomyStore
- Created `src/db/sync/syncInventory.ts` with two exports:
  - `upsertInventory(authId, { xp, coins, gems })` — updates user_profiles row by authId with current economy values
  - `fetchInventory(authId)` — selects xp, coins, gems from user_profiles by authId, returns object or null
- Note: XP/coins/gems live on `user_profiles` table (not `inventory` table — that's for items like chests/boosters)
- Used `db.update()` instead of upsert since the user profile row should already exist (created during sign-in via US-004)
- Hooked into `useEconomyStore.addXP`, `addCoins`, `addGems` — after `set()`, reads updated state via `get()` and fires upsertInventory
- authId sourced from `useAuthStore.getState().email` (same pattern as US-004/US-005)
- No new TS errors introduced
- Files affected: `src/db/sync/syncInventory.ts` (new), `src/features/economy/useEconomyStore.ts` (modified), `src/features/database/PRD35_NeonDBIntegration.md` (updated)
- Learnings for next cycles:
  - Patterns: Economy values (xp, coins, gems) are columns on user_profiles, not on the inventory table
  - Patterns: `db.update().set().where()` is simpler than upsert when the row is guaranteed to exist
  - Patterns: After Zustand `set()`, `get()` immediately reflects the new state (synchronous)
  - Traps: Don't confuse `inventory` DB table (items) with economy/currency values (on user_profiles)
  - Helpful context: spendCoins/spendGems not synced — could add later if needed for consistency
