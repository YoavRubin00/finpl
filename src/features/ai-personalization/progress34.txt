## Iteration 7 — US-004 AC#2: Track monetization signals in useAITelemetryStore
- Added 5 new monetization event types to `TelemetryEventType`: `shop_browse`, `hearts_depleted`, `chest_unlock`, `iap_dismissed`, `iap_accepted`
- Added `MonetizationSignalType` utility type (Extract of monetization events from TelemetryEventType)
- Added `trackMonetizationSignal(signal, meta?)` convenience method to `useAITelemetryStore` — uses moduleId `_monetization` and sets `isMonetizationSignal: true` in meta
- Enhanced `deriveMonetizationVector()` in `analyzeProfile.ts` — now prioritizes behavioral signals over completion-speed heuristic:
  - `shopBrowses >= 3` or `chestUnlocks >= 2` → Impulse Buyer
  - `iapAccepts >= 1` + high accuracy → Status Seeker
  - `heartsDepleted >= 2` or hearts+dismissals → Anxious/Needs Security
  - `iapDismissals >= 3` with no accepts → Frugal/Value Driven
  - Falls back to original completion-speed heuristic when no monetization signals exist
- Files affected: `src/features/ai-personalization/types.ts`, `src/features/ai-personalization/useAITelemetryStore.ts`, `src/features/ai-personalization/analyzeProfile.ts`
- Learnings for next cycles:
  - `trackMonetizationSignal` is the single entry point for all monetization telemetry — call from shop screen, heart depletion handler, chest UI, IAP modals
  - moduleId `_monetization` is a sentinel value — won't collide with real module IDs
  - `MonetizationSignalType` can be imported by UI components for type-safe signal tracking
  - Monetization signals now take priority over speed heuristics — behavioral data is stronger signal
  - Next task: US-004 AC#3 — Create a DynamicIAPService that customizes IAP copy/timing based on AI profile

## Iteration 8 — US-004 AC#3: DynamicIAPService
- Created `DynamicIAPService.ts` — stateless service functions (no Zustand store needed)
- Core API: `evaluateIAPOffer(trigger: MonetizationTrigger): IAPDecision`
  - Reads `profile.monetizationContext.offers` from `useAITelemetryStore.getState()`
  - Finds exact-match offer for the given trigger
  - Returns `shouldShow`, `offer`, `displayDurationMs`, `entryDelayMs`
- Anti-spam: 5-minute cooldown + max 3 offers per session
- Urgency-based timing: high urgency = 500ms delay / 8s display, low = 2.5s delay / 5s display
- Helper functions: `recordOfferShown()`, `recordOfferAccepted(bundleKey)`, `recordOfferDismissed(bundleKey)`
- Accept/dismiss funcs log to telemetry via `trackMonetizationSignal` — closes the feedback loop
- `resetIAPSession()` for app foreground / new session
- Files affected: `src/features/ai-personalization/DynamicIAPService.ts` (new), `src/features/ai-personalization/PRD34_AIPersonalization.md` (updated)
- Learnings for next cycles:
  - `evaluateIAPOffer` is the single entry point — call from quiz failure handler, chapter complete, heart depletion, etc.
  - Session state (lastShownTimestamp, offersShownThisSession) is module-scoped, not persisted — resets on app restart which is desired
  - `useAITelemetryStore.getState()` used outside React (pure function, no hook) — valid Zustand pattern
  - Accept/dismiss recording feeds back into `deriveMonetizationVector` on next analysis cycle
  - Future UI: create an `IAPPopupModal` component that consumes the `IAPDecision` return value
  - All PRD 34 tasks are now complete
